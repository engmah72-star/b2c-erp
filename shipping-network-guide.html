📘 دليل شبكة الشحن V2

المعمارية، الـ workflows، الـ events، الصلاحيات

🎯 الفلسفة: هذه ليست منظومة شحن لشركة كروت شخصية — دي منصة لوجستية وطنية تربط أي شركة شحن، كابتن مستقل، أو فرع امتياز بأي عميل في أي محافظة، بنموذج Marketplace كامل (commission + escrow + SLA + ratings).

النظام القديم (shipping.html + 5 صفحات) يفضل شغّال لـ backward compatibility (RULE 6). النظام الجديد parallel — كل أوردر جديد يقدر يفتح بالـ V2.

Layer 3 — Financial Integration

كل event مالي يمر عبر financial-sync-engine.js ويُسجَّل في financial_ledger (RULE 5).

Eventالأثر المالي
PARTNER_ONBOARDEDإنشاء محفظة للشريك (balance=0)
SHIPMENT_DELIVERED+ accrual للشريك (pendingPayout) + accrual لعمولة المنصة
PARTNER_PAYOUT- من محفظة المنصة، + لمحفظة الشريك (atomic batch)
SLA_PENALTYخصم من pendingPayout + + لمحفظة المنصة
PLATFORM_SHIPPING_COMMISSIONincome event للمنصة

🚦 Shipment Lifecycle (State Machine)

created ← assigned ← accepted ← picked_up ← in_transit ← out_for_delivery ← delivered ← settled

State machine validation في shipping-network-core.js — أي انتقال غير مسموح يُرفض.

📐 سير العمل (Workflows)

1. إضافة شريك جديد (Onboarding)

  1. Admin/Dispatcher يفتح shipping-partners.html
  2. يضغط "➕ شريك جديد" ويملأ النموذج
  3. الـ Engine يُنشئ بشكل atomic: shipping_partners doc + wallets doc للشريك
  4. الحالة الأولية: kycStatus=pending
  5. Admin يراجع البيانات، ويضغط ✅ اعتماد KYC → الحالة تصبح approved
  6. الشريك الآن قابل لاستلام شحنات

2. إنشاء شحنة وإسنادها

  1. من shipping-dispatch.html اضغط "➕ إنشاء شحنة"
  2. اختر الأوردر + المحافظة + الوزن + طريقة الدفع
  3. اختر نوع الإسناد:
    • Manual — الـ Dispatcher يختار من قائمة الشركاء المُرشَّحين
    • Auto Route — النظام يختار أفضل شريك (rating + onTimePct + price + coverage)
    • Bid (Marketplace) — الشحنة تتطرح للعروض، الشركاء يقدموا عروضهم
  4. الشريك المختار يستلم notification (Phase 2+)
  5. السائق يفتح driver-app.html ويبدأ الـ pickup

3. التسليم وإثبات الـ POD

  1. السائق يضغط "📍 وصلت للمنطقة"
  2. يضغط "✓ تم التسليم"
  3. يدخل اسم المستلم + ملاحظات + صورة (POD)
  4. الـ GPS يُسجَّل تلقائياً
  5. الـ Engine يحسب SLA → لو متأخر، يُسجَّل في sla_breaches لمراجعة admin
  6. يُحسب pendingPayout للشريك (cost) + accrual لعمولة المنصة

4. التسوية مع الشريك (Payout)

  1. من shipping-partners.html → تفاصيل الشريك → ترى pendingPayout
  2. من shipping-accounts.html القديمة (V1) أو حسابات منفصلة → اصرف للشريك
  3. الـ Engine يحرك المبلغ من محفظة المنصة → محفظة الشريك (atomic batch)
  4. يُسجَّل في financial_ledger كـ MERCHANT_PAYOUT

5. غرامات SLA

  1. من shipping-sla.html → تبويب "خرق SLA"
  2. راجع الـ breaches "pending_review"
  3. إما تطبق غرامة (تخصم من pendingPayout) أو تعفو

📊 اختبار 6 أسئلة الحوكمة

السؤالالإجابة
قابل للتوسع جمهورياً؟✅ نعم — governorateId + cityId في كل entity
يقلل الاعتماد الداخلي؟✅ نعم — أي شركة شحن تنضم بدون كود
يزيد قوة الشبكة؟✅ نعم — كل partner = توسع تغطية
يحتفظ بالبيانات؟✅ نعم — كل tracking في shipment_events
المنصة مركز التحكم؟✅ نعم — commission + escrow + SLA enforcement
قابل لـ Marketplace؟✅ نعم — shipment_bids جاهز

🛡️ القواعد المُطبَّقة

  • RULE 1 — Single Source of Truth: الأرصدة في wallets فقط. الشريك يقرأ pendingPayout (denormalized cache) لكن المصدر الحقيقي الـ ledger.
  • RULE 2 — Event Driven: كل عملية عبر dispatchShippingNetworkEvent().
  • RULE 3 — Atomic Writes: كل handler في الـ engine يستخدم writeBatch.
  • RULE 4 — No Module Owns Money: الـ partner يقرأ رصيد محفظته، لا يكتبها. الـ engine بس يكتب.
  • RULE 5 — Full Audit: كل state change → shipment_events. كل event مالي → financial_ledger.
  • RULE 6 — Backward Compat: 6 صفحات شحن قديمة تفضل شغّالة بدون أي تعديل.
  • RULE 8 — Data Boundaries: الشريك يرى شحناته فقط (عبر isOwnPartnerData()). السائق يحدّث شحناته فقط.
